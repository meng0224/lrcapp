# 後續路線建議

本文件是 LrcApp 在 [DEVELOPMENT_ROADMAP.md](./DEVELOPMENT_ROADMAP.md) Phase 1–4 之後的策略性發展方向建議。定位上與現有路線圖互補：路線圖追蹤「正在做什麼」，本文件討論「接下來值得做什麼、為什麼」。

建議按優先級排列，每項標示前置條件與預期效益。

---

## 0. 前提：Phase 1–4 收尾要點

以下阻塞項在進入任何新方向前應先解決：

| 阻塞項 | 影響 | 建議行動 |
| --- | --- | --- |
| Gradle wrapper 無法下載 `gradle-9.0-milestone-1-bin.zip` | 所有測試無法在本地跑通 | 降級至穩定版（如 `8.10`），確保 CI 與本機都能建置 |
| 單元測試從未實際執行 | Phase 1/3 的修正缺乏回歸保護 | 解決 Gradle 後第一時間跑 `testDebugUnitTest` 並修復紅燈 |
| Android 11+ 實機驗證未完成 | SAF 授權、遞迴匯入、來源目錄輸出的正確性不確定 | 至少在一台 Android 11+ 實機或模擬器上完成 PHASE2 驗證 |
| `MainActivity.kt`（1151 行）職責過重 | 任何功能新增都會加劇耦合，Phase 4 的拆分是後續所有工作的基礎 | 在開始新功能前完成 Phase 4 |

### Phase 4 架構重整的具體切入建議

Phase 4 目前只列了目標，以下是建議的實作順序：

1. **引入 ViewModel**：把 `files: MutableList<SubtitleFile>`、`settings: AppSettings`、轉換進度等狀態搬進 `MainViewModel`，用 `StateFlow` 取代直接操作 adapter。這一步不改 UI，只拆狀態。
2. **抽出 ConversionUseCase**：把 `startConversion()` 內的迴圈、狀態更新、錯誤處理搬成獨立類別，ViewModel 只負責觸發與觀察。
3. **抽出 StorageRepository**：把 `StorageHelper` + `SettingsManager` 包成 repository 層，統一對外介面，讓 ViewModel 不直接碰 `DocumentFile` 或 `SharedPreferences`。
4. **SubtitleConverter 拆分**：目前所有格式解析放在同一個類別。改為每個格式一個 `SubtitleParser` 介面實作（`VttParser`、`AssParser`、`SrtParser` 等），由 `SubtitleConverter` 依副檔名分派。這會讓新增格式變成只加一個檔案。
5. **統一錯誤模型**：目前錯誤表達散落在 `null`、`Pair<Boolean, String?>`、`FileStatus.ERROR`。改用 `sealed class ConversionResult` / `SaveResult` 統一，讓呼叫端可以用 `when` 窮舉處理。

## 1. 工程基礎設施（建議優先級：高）

### 1.1 CI/CD 管線

目前專案沒有任何自動化管線，所有測試與建置都靠手動。

**建議**：建立 GitHub Actions workflow，至少涵蓋：

```
push / PR → build → unit test → lint → (optional) instrumentation test
```

- 基本的 build + unit test 只需一個 workflow 檔，5 分鐘內可完成
- Instrumentation test 可用 GitHub Actions 的 Android emulator action，但因速度較慢，可設為手動觸發或只在 main 分支跑
- 加入 test 通過作為 PR merge 的必要條件

**效益**：防止回歸、減少手動驗證負擔、讓 Phase 3 的測試投資真正發揮作用。

### 1.2 Gradle 版本穩定化

目前使用 `gradle-9.0-milestone-1-bin.zip`（里程碑版），導致部分環境無法下載。

**建議**：降級至最新穩定版（如 `8.10` 或當時的最新穩定版），並在 CI 中鎖定版本。

### 1.3 依賴版本更新

目前依賴版本已有一段時間未更新（如 Material `1.11.0`、Lifecycle `2.7.0`）。

**建議**：
- 導入 Gradle Version Catalog（`libs.versions.toml`）集中管理版本
- 定期檢查 `./gradlew dependencyUpdates`（需加 plugin）
- 特別注意 `compileSdk` / `targetSdk` 隨 Google Play 政策要求提升

## 2. 功能擴展（建議優先級：中）

以下功能建議依使用者價值排序，每項獨立可做，不互相依賴。

### 2.1 歌詞預覽

**現狀**：轉換完成後使用者只看到「成功」狀態與輸出檔名，無法確認內容是否正確。

**建議**：在檔案列表的成功項目上，點擊可展開或彈出 bottom sheet 顯示 LRC 內容預覽。不需要播放功能，純文字即可。

**前置條件**：Phase 4 完成後，預覽邏輯放在 ViewModel 層，不會再加重 MainActivity。

### 2.2 新格式支援

**目前支援**：VTT、ASS/SSA、SRT、STR、SMI、SUB → LRC

**可考慮新增**：

| 格式 | 說明 | 難度 |
| --- | --- | --- |
| TTML / DFXP | Netflix、YouTube 常用的 XML 字幕 | 中（XML 解析） |
| SBV | YouTube 下載字幕的常見格式 | 低（類似 SRT） |
| TXT（純時間戳） | 部分音樂軟體匯出的簡易格式 | 低 |

**前置條件**：SubtitleConverter 拆分完成後，新增格式只需實作一個 `SubtitleParser`。

### 2.3 反向轉換（LRC → 其他格式）

**場景**：使用者已有 LRC 檔，需要轉成 SRT 或其他格式供影片軟體使用。

**建議**：這是一個可以顯著擴大使用者群的功能，但需要重新設計 UI 的輸入/輸出格式選擇流程。建議在架構穩定後再評估。

### 2.4 Smart Naming 與 Time Precision 的 UI 開關

**現狀**：這兩個功能固定啟用，`AppSettings` 已有欄位但 UI 沒有對應開關。

**建議**：在儲存位置卡片區域加入進階設定展開面板。改動量小，但能提升進階使用者的控制感。

## 3. 使用者體驗改善（建議優先級：中）

### 3.1 空狀態引導

**現狀**：App 啟動後是空列表，使用者需要自己理解操作流程。

**建議**：在空列表狀態顯示引導圖示與文字，說明「選擇字幕檔開始轉換」，並根據目前輸出模式顯示對應提示。

### 3.2 轉換結果摘要

**現狀**：轉換完成後用 Snackbar 顯示成功/失敗數量。

**建議**：改用 dialog 或展開式卡片顯示完整摘要，包含：
- 成功數量與輸出位置
- 失敗數量與各失敗原因分類
- 可一鍵開啟輸出目錄（如果系統支援）

### 3.3 深色模式驗證

**現狀**：`values-night/colors.xml` 存在，但未見完整的深色模式測試記錄。

**建議**：在 CI 的 instrumentation test 中加入深色模式截圖測試，確保 Material 3 主題在兩種模式下都正常。

## 4. 技術債與長期演進（建議優先級：低）

### 4.1 Compose 遷移評估

**現狀**：目前使用 View + XML，Material 3 主題已套用。

**建議**：
- 短期內不建議遷移。目前 UI 只有一個頁面，View-based 的維護成本可控。
- 當出現以下訊號時再評估：需要加入第二個頁面、需要複雜的動態 UI（如拖曳排序）、或 Google 停止維護 View-based Material Components。
- 若決定遷移，建議採漸進策略：先在 `ComposeView` 中嵌入新元件，不一次性重寫。

### 4.2 多模組拆分

**現狀**：單一 `app` 模組。

**建議**：
- 當 SubtitleConverter 拆分完成且有明確的 parser 介面後，可考慮將轉換引擎抽成獨立 `:converter` 模組。
- 效益：獨立測試、build 快取更有效、未來可供其他 app 或 CLI 工具重用。
- 目前規模下不急，但在 Phase 4 拆分時可預留模組邊界。

### 4.3 Play Store 上架準備

若未來考慮上架，需要處理：

| 項目 | 現狀 | 需要做的 |
| --- | --- | --- |
| `applicationId` | `com.example.lrcapp` | 改為正式 ID（如 `com.yourname.lrcapp`） |
| `versionCode` / `versionName` | `1` / `"1.0"` | 建立版本管理策略 |
| 簽名設定 | 未設定 release signing | 建立 keystore 與 signing config |
| ProGuard / R8 | `minifyEnabled false` | 啟用並測試混淆 |
| 隱私權政策 | 無 | Play Store 要求必須有 |
| 目標 SDK | 34 | 需隨 Google 政策每年更新 |
