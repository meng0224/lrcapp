# LrcApp Figma 主提示詞

本文件提供一段可直接貼進 Figma AI 或同類設計生成工具的單一主提示詞。內容以目前 repo 的真實功能、現有 Android 畫面結構與互動細節為準，目標是優化主畫面體驗，而不是擴充產品功能。

## 使用方式

- 直接複製下方「主提示詞」整段貼入 Figma AI。
- 若需要追加限制，優先在提示詞最後補充，不要改動中段的產品邏輯描述。
- 這份提示詞預設目標是 Android 手機版高保真主畫面與狀態變體。

## 主提示詞

```text
你是一位資深 Android 產品設計師，請為一個 Android 原生工具型 App 設計高保真手機介面。產品名稱是 LrcApp，功能是把常見字幕格式批次轉換成 LRC 歌詞格式。

請先理解產品定位：這不是音樂播放器，不是歌詞編輯器，也不是多步驟精靈，而是一個單頁的批次字幕轉檔工作台。整體體驗要偏向成熟、可靠、效率導向的檔案處理工具。請保留 Android / Material 的可用性基礎，但不要做成制式的紫色 demo 風格，也不要有播放器、潮流音樂 App、霓虹娛樂化語言。

這個產品目前的核心流程固定是：
設定輸出位置 -> 選擇文件或資料夾 -> 驗證 -> 開始轉換 -> 自動保存

以下是不可改動的產品邏輯，請嚴格遵守，不要自行新增功能或改流程：

1. 這是一個 Android 單頁工具，請維持單一主畫面，不要改成多頁流程，不要拆成 wizard，不要新增主要分頁。
2. 支援輸入格式：VTT、ASS、SSA、SRT、STR、SMI、SUB。輸出格式固定為 LRC。
3. 一般模式下，使用者透過系統文件選擇器一次選擇多個字幕檔。
4. 若開啟「遞迴匯入子資料夾字幕」，主 CTA 會從「選擇文件」變成「選擇資料夾」，並遍歷子資料夾匯入字幕檔。
5. 驗證規則固定存在：不支援格式要標示為無效，單檔超過 10MB 也要標示為無效。
6. 有三種輸出模式，而且必須讓使用者清楚知道目前輸出到哪裡：
   - 預設應用下載目錄
   - 自訂 SAF 授權目錄
   - 輸出到原文件目錄
7. 「自訂 SAF 授權目錄」與「輸出到原文件目錄」互斥，不能同時啟用。
8. 若啟用「輸出到原文件目錄」，第一次寫入某個來源資料夾時需要授權；若是遞迴匯入，會先對匯入根目錄授權，之後保存時沿用同一授權並重建原始子資料夾結構。
9. 一般模式下，第二次選檔會覆蓋舊列表；原文件目錄模式下，第二次選檔會追加到現有列表；遞迴匯入也會追加並去除重複 Uri。
10. 遞迴匯入時，不支援格式或超過 10MB 的檔案會直接略過，不加入列表。
11. 轉換完成後會自動保存所有成功檔案，不提供單檔下載、ZIP 打包、歌詞預覽、雲端同步或批次設定面板。
12. 若保存時失敗，例如 SAF provider 改寫副檔名導致結果不是有效的 .lrc，介面不能誤報成功，必須清楚呈現保存失敗。
13. smart naming 與 time precision 目前固定啟用，但沒有 UI 開關，請不要設計這兩個設定入口。

請以「主畫面優化」為目標，重整目前的資訊層級、版面結構、CTA 優先順序、狀態視覺與回饋方式，但不要改變任務結構。現有畫面特徵包括：

- 頂部有 App bar，標題是 LrcApp，副標是「批次字幕轉 LRC 工作台」。
- 主內容區是一個可滾動的單頁版面。
- 上方會有「儲存位置」卡片，裡面顯示目前輸出模式、輸出位置、切換到原文件目錄的 switch、遞迴匯入的 switch，以及自訂目錄相關操作。
- 在等待來源目錄或匯入根目錄重新授權時，畫面上方會出現 inline 授權提示 banner 與「授權目錄」按鈕。
- 畫面中段有文件列表區，空態時顯示支援格式與輸出格式說明；有資料時顯示可轉換數量與檔案列表。
- 文件列表 item 至少包含：檔名、檔案大小、格式 badge、狀態 chip、成功時的輸出檔名、失敗或無效時的錯誤訊息。
- 列表狀態包含：待處理、無效、處理中、成功、失敗。
- 轉換進行中會出現進度卡片，顯示進度條、目前處理文案、以及像 3/12 這種計數。
- 底部有固定的 action card，裡面有主要 CTA 與次要操作。空態時主 CTA 是「選擇文件」或「選擇資料夾」；有檔案時主 CTA 會變成「開始轉換 (n)」；次要按鈕包含「重新選擇 / 新增文件 / 新增資料夾」與「清除列表」。
- 系統回饋除了 inline banner，也會使用 Snackbar 或輕量提示。

請至少設計以下主畫面狀態：

1. 預設空態主畫面
2. 已設定自訂輸出目錄的主畫面
3. 已開啟「輸出到原文件目錄」的主畫面
4. 已開啟「遞迴匯入子資料夾字幕」的主畫面
5. 文件列表混合狀態，需同時出現待處理、無效、處理中、成功、失敗
6. 轉換進度中的狀態
7. 等待來源目錄或匯入根目錄授權的提示狀態
8. 轉換完成後部分保存成功、部分保存失敗的狀態
9. 沒有可轉換文件時，主 CTA disabled 的狀態

請務必包含並定義這些元件與互動：

- App bar
- 儲存位置卡片
- 模式 chip 或 badge
- Switch
- 授權提示 banner
- Progress card 與 progress indicator
- 文件列表 item
- Status chips：待處理、無效、處理中、成功、失敗
- 底部固定 action card
- 主要 CTA、次要 CTA
- Snackbar / inline feedback / dialog 建議

視覺與體驗要求：

- 專業工具感、可信、乾淨、成熟
- 高掃讀性，資訊密度適中，不要過度留白導致操作分散
- 儲存位置、目前模式、下一步操作必須一眼看懂
- 列表中的狀態必須 1 秒內可辨識
- 成功、處理中、失敗、警示要明確區隔，但避免刺眼高飽和配色
- 風格要符合 Android 原生使用情境與 Material Components 語境
- 請使用繁體中文介面文案

可以沿用以下現有視覺方向作為基礎再優化，不要偏離太遠：

- 主色偏沉穩藍色系，例如 brand primary 約為 #1976D2，深色藍約為 #0D47A1，淺藍底約為 #E3F2FD
- 背景是淺灰藍白系，不是純白，例如 surface 約為 #F3F6F8、card 約為 #FFFFFF
- 文字以深灰藍色為主，例如 primary text 約為 #24323D、secondary text 約為 #61717D
- 成功為柔和綠色系，失敗為柔和紅色系，警示為暖橘色系，不要用螢光色
- 卡片圓角偏 16dp，底部 action card 可以更大一些，例如 24dp

請一併交付：

1. 1 個 Android 手機尺寸的高保真主畫面設計
2. 8 個以上同畫面的關鍵狀態變體
3. 1 組小型 design system / style tile
4. 主要元件規格，包含卡片、按鈕、switch、status chip、banner、列表 item
5. 色彩系統、字級層級、間距規則
6. 重要互動狀態說明，包含 default、pressed、disabled、processing、success、error
7. 為什麼這樣安排資訊層級、CTA 優先序與回饋方式的設計 rationale

請避免以下錯誤方向：

- 不要做成音樂播放器 UI
- 不要做成多步驟精靈頁
- 不要新增歌詞預覽、編輯器、ZIP 匯出、雲端同步、進階設定頁
- 不要隱藏關鍵操作
- 不要讓視覺造型凌駕於可讀性與效率
- 不要把它設計成桌面檔案管理器移植版，而是 Android 原生工具 App
```
