# LrcApp Figma 設計提示詞包

本文件整理一組可直接交給 Figma 設計師或設計型 AI 的中文提示詞，目的不是重做整個產品，而是優化 LrcApp 目前的 Android 主畫面體驗。

## 使用方式

- 如果你要一次交付完整需求，直接使用「主提示詞」。
- 如果設計師要分段產出，使用後面的 4 組子提示詞。
- 若要給 Figma AI、Lovable、v0 類工具，建議先貼「主提示詞」，再補貼對應子提示詞。

## 1. 主提示詞

```text
你是一位資深 Android 產品設計師，請為一個 Android 原生工具型 App 設計高保真手機介面。產品名稱是 LrcApp，功能是把常見字幕格式批次轉換成 LRC 歌詞格式。

請注意，這不是音樂播放器，也不是歌詞編輯器，而是一個偏專業工具感的批次轉檔工作台。設計方向要強調效率、可信度、清楚的狀態可視化、低學習成本，以及 Android 使用者熟悉的互動邏輯。請保留 Android / Material 的可用性基礎，但不要做得過於制式或無聊；整體氣質要像一個乾淨、成熟、可靠的檔案處理工具。

產品條件與不可改動的邏輯如下：
1. 這是一個 Android 單頁工具，不要改成多步驟 wizard，也不要拆成多個主要頁面。
2. 核心流程固定為：選擇輸出方式 -> 選擇文件 -> 驗證文件 -> 開始轉換 -> 自動保存。
3. 支援的輸入格式有 VTT、ASS、SSA、SRT、STR、SMI、SUB，輸出格式固定是 LRC。
4. 文件選擇使用系統文件選擇器，可一次選擇多個文件。
5. 驗證規則固定存在：不支援格式要標示為無效，單檔超過 10MB 也要標示為無效。
6. 有三種輸出模式：
   - 預設應用下載目錄
   - 自訂 SAF 授權目錄
   - 輸出到原文件目錄
7. 「自訂輸出目錄」與「輸出到原文件目錄」互斥，不能同時啟用。
8. 若啟用「輸出到原文件目錄」，第一次寫入某個來源資料夾時，系統會要求使用者逐目錄授權；同一來源資料夾之後會重用已保存授權。
9. 一般模式下，第二次選檔會覆蓋舊列表；原文件目錄模式下，第二次選檔會追加到現有列表，並且要對重複文件去重。
10. 轉換完成後會自動保存所有成功檔案，不提供 ZIP 打包、單檔下載、歌詞預覽、雲端同步或批次設定面板。
11. 目前沒有 smart naming 與 time precision 的 UI 開關，不要自行設計這兩個設定入口。

請以「主畫面優化」為目標，重新整理現有單頁介面的資訊層級、版面結構、CTA 優先順序、卡片設計、狀態元件與回饋訊息。可以改進視覺與互動提示，但不要改變產品任務結構。

請產出以下內容：
1. Android 手機尺寸高保真主畫面設計
2. 同一畫面的多個狀態版本
3. 可重複使用的 UI 元件規格
4. 視覺風格說明與設計 rationale

請至少設計這些畫面狀態：
1. 預設狀態主畫面
2. 已設定自訂輸出目錄的主畫面
3. 已開啟「輸出到原文件目錄」的主畫面
4. 空列表狀態
5. 文件列表混合狀態：待處理、無效、處理中、成功、錯誤同時出現
6. 轉換進度中的狀態
7. 來源目錄授權引導或等待授權的提示狀態

請務必包含以下元件：
- App bar
- 儲存位置卡片
- 主要 CTA 與次要 CTA
- Switch
- Progress indicator
- 文件列表 item
- Status chips：待處理、無效、處理中、成功、錯誤
- toast / snackbar / inline feedback 建議

視覺語言要求：
- 專業工具感、可信、乾淨、資訊密度適中
- 以可掃讀性與操作效率為優先
- 可使用更成熟的配色與層級，不要停留在預設紫色 Android demo 風格
- 避免過度娛樂化、播放器感、霓虹感、潮流音樂 App 語言
- 避免只靠大面積留白造成資訊分散
- 列表與狀態需要一眼可判讀
- 要讓使用者理解「現在輸出到哪裡」、「哪些文件可轉換」、「哪些需要處理或重選」

請附帶定義：
- 色彩系統
- 字級層級
- 間距規則
- 卡片、按鈕、狀態 chip 規格
- 重要互動狀態（default / pressed / disabled / error / success）

請避免以下錯誤方向：
- 不要做成音樂播放器 UI
- 不要改成多步驟精靈頁
- 不要新增專案裡不存在的功能
- 不要讓造型感壓過可讀性與功能效率
- 不要把關鍵操作藏得太深

請以繁體中文介面文案為主，並遵守 Android 原生使用情境與 Material 慣例。
```

## 2. 子提示詞 A: 主畫面版型與資訊層級

```text
請基於以下產品條件，設計 LrcApp 的 Android 主畫面高保真稿，重點是版面結構與資訊層級，而不是額外加功能。

產品是一個字幕轉 LRC 的批次工具。核心流程是：選擇輸出方式 -> 選擇文件 -> 驗證 -> 轉換 -> 自動保存。整體介面維持單頁，不改成多頁流程。

請你優化以下區塊的順序與層級：
- App bar
- 儲存位置區塊
- 選擇文件 / 開始轉換兩個主要操作
- 進度區塊
- 文件列表

請特別解決這些問題：
- 使用者需要快速理解目前輸出到哪裡
- 使用者需要快速理解接下來第一步該做什麼
- 主要 CTA 與次要操作不能互搶注意力
- 畫面在沒有文件、有很多文件、轉換中時都要維持清楚

請產出：
1. 主畫面整體 layout
2. 區塊層級說明
3. Android 手機尺寸高保真稿
4. 設計 rationale，說明為何這樣安排資訊優先序

風格關鍵字：專業工具感、成熟、清楚、效率導向、可信、易掃讀。
```

## 3. 子提示詞 B: 儲存位置卡片與輸出模式切換

```text
請專注設計 LrcApp 的「儲存位置卡片」與相關互動狀態。這個區塊是整個產品最重要的理解成本來源，因為使用者必須明白輸出結果會保存到哪裡。

已知邏輯如下：
- 有三種輸出模式：預設應用下載目錄、自訂 SAF 授權目錄、輸出到原文件目錄
- 自訂目錄與原文件目錄模式互斥
- 原文件目錄模式下，第一次寫入某來源資料夾時需要逐目錄授權，後續可重用授權

請你設計：
1. 預設下載目錄狀態
2. 已選擇自訂 SAF 目錄狀態
3. 已開啟原文件目錄模式狀態
4. 原文件目錄模式下的授權提示樣式
5. 次要操作按鈕，例如「選擇目錄」、「清除目錄」、「清除文件列表」如何與主流程共存

設計目標：
- 一眼看懂目前輸出位置
- 一眼看懂哪個模式正在生效
- 互斥邏輯要清楚，但不要做得像設定頁
- 授權這件事要被說明清楚，但不要造成恐嚇感

請附上：
- 卡片結構
- 文案語氣
- 狀態樣式
- switch、輔助說明文字、按鈕的關係
- 錯誤或受限狀態的呈現建議
```

## 4. 子提示詞 C: 文件列表卡片與狀態系統

```text
請為 LrcApp 設計文件列表 item 與狀態系統。這是使用者判斷哪些文件可轉換、哪些失敗、哪些已成功保存的主要工作區。

每個文件 item 至少有：
- 文件名
- 狀態
- 成功時的輸出檔名
- 失敗或無效時的錯誤訊息

目前狀態包含：
- 待處理
- 無效
- 處理中
- 成功
- 錯誤

請你設計：
1. 單個文件 item 的視覺結構
2. 狀態 chip / badge 的樣式系統
3. 成功、錯誤、無效訊息的層級差異
4. 多筆列表同時混合不同狀態時，如何保持可掃讀性
5. 長檔名、長錯誤訊息、輸出檔名顯示的處理方式

設計要求：
- 優先提升掃讀效率，不要過度裝飾
- 狀態要在 1 秒內可辨識
- 成功與錯誤要足夠區隔，但不要靠刺眼高飽和色硬撐
- item 要適合 Android 工具型 app，而非郵件或聊天 UI

請附上：
- item 元件規格
- 狀態色與文字規格
- 列表間距與密度建議
- 混合狀態畫面示意
```

## 5. 子提示詞 D: 空態、錯誤態、進度與授權引導

```text
請為 LrcApp 補齊主畫面中的關鍵狀態稿，重點是讓使用者在工具型流程中知道現在發生什麼事，以及下一步該做什麼。

需要設計的狀態包括：
1. 空列表狀態
2. 已選檔但尚未轉換
3. 轉換進行中
4. 轉換完成，部分成功部分失敗
5. 沒有可轉換文件時的提示
6. 原文件目錄模式下，等待來源目錄授權的提示
7. 因輸出模式互斥而無法執行某操作時的回饋

設計目標：
- 讓工具狀態透明，不要讓使用者猜
- 提示下一步行動
- 減少對系統權限與 SAF 授權的困惑
- 優先使用輕量但明確的引導，例如 inline hint、snackbar、dialog、bottom sheet

限制條件：
- 仍然是單頁工具體驗
- 不新增新的主流程頁面
- 不設計歌詞預覽、編輯器、雲端同步、ZIP 下載

請輸出：
- 每種狀態的高保真畫面
- 文案與互動建議
- 哪些狀態適合用 snackbar，哪些應該用 inline message 或 dialog
- 進度條、完成提示、授權提醒的視覺語言建議
```

## 6. 交付檢查清單

把提示詞交給設計師前，確認至少包含以下要求：

- 已說明這是 Android 原生單頁工具，不是播放器或多步驟流程
- 已說明產品核心流程與不可變的業務邏輯
- 已明確列出 3 種輸出模式與互斥條件
- 已要求設計空態、進度態、錯誤態、成功態與授權引導
- 已要求定義元件與設計系統，而不是只交一張漂漂亮亮的首頁
- 已禁止加入專案裡不存在的功能
- 已要求使用繁體中文介面文案

## 7. 建議交付物格式

如果你要向設計師明確要求產出，可以在提示詞最後再補一段：

```text
請最後交付以下內容：
1. 1 個 Android 手機主畫面高保真 frame
2. 6 到 8 個關鍵狀態變體
3. 1 組小型 design system / style tile
4. 主要元件規格與註解
5. 重要互動與文案決策說明
```
